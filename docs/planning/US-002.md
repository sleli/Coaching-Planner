## User Story

**Epic:** EP-001 — Anagrafiche e struttura operativa
**Priorità:** HIGH | **Story Points:** 3

**Story**
As Marta,
I want creare, aggiornare e chiudere commesse associate a un cliente,
so that possa organizzare il lavoro per incarico e distinguere ciò che è ancora operativo da ciò che è concluso.

**Criteri di Accettazione**
- [ ] Marta può creare una commessa associandola a un cliente attivo e la commessa compare nell'elenco del cliente corretto.
- [ ] Marta può aggiornare i dati principali di una commessa esistente senza perderne l'associazione al cliente.
- [ ] Marta può chiudere una commessa e la commessa non appare più tra quelle operative per nuovi inserimenti.

---

## Analisi dei Requisiti

> **Analista:** Emanuele 🔎

### Scope

- Introdurre un nuovo modello di dominio `Project` (commessa) associato a un `Client` esistente, nel backoffice protetto.
- Consentire creazione, aggiornamento dati principali e chiusura logica delle commesse.
- Validare che la commessa sia associata a un cliente attivo in fase di creazione.
- Mostrare in elenco lo stato aperto/chiuso della commessa e il cliente associato.
- Preparare query riusabili per filtrare solo commesse operative, da utilizzare nelle story successive (US-003, US-004).
- Resta fuori scope: budget ore (US-003), assegnazione collaboratori (US-004), cancellazione fisica, import/export, storicizzazione modifiche.

### Requisiti Impliciti

- Serve un nuovo modello Prisma `Project` con relazione many-to-one a `Client`.
- La chiusura deve essere logica (flag persistente), non cancellazione fisica.
- Dati minimi della commessa: nome (obbligatorio), cliente associato (obbligatorio, attivo) e stato aperto/chiusa.
- La creazione deve validare che il `clientId` esista e che il cliente sia attivo.
- Il nome commessa deve essere univoco all'interno del cliente (stessa logica anti-duplicato applicativa di US-001).
- La logica server deve esporre `listActiveProjects()` per i futuri flussi operativi.
- La navigazione laterale del backoffice deve includere la voce "Commesse" come previsto dal mockup.

### Assunzioni

- I dati principali della commessa per questa story sono: `name`, `clientId`, `isActive`. Eventuali campi aggiuntivi (budget ore, date, descrizione) saranno introdotti da US-003 e story successive.
- Il nome commessa deve essere univoco all'interno dello stesso cliente (coppia `clientId` + `normalizedName` unica).
- La UI di creazione/modifica userà un selettore a discesa con la lista dei soli clienti attivi (riusando `listActiveClients()`).
- Il mockup di riferimento per la sidebar è `docs/mockups/dashboard_backoffice_soci/code.html`; per la pagina elenco si segue lo stesso pattern visivo della pagina clienti già implementata.

### Modifiche al Data Model

- Aggiungere modello `Project` con:
  - `id` — UUID come chiave primaria
  - `name` — String, obbligatoria
  - `normalizedName` — String, univoca in combinazione con `clientId`
  - `isActive` — Boolean, default `true` (aperta). `false` = chiusa.
  - `clientId` — String, FK → `Client.id`, obbligatoria
  - `createdAt` — DateTime, default `now()`
  - `updatedAt` — DateTime, auto-update
- Relazione: `Client` hasMany `Project`, `Project` belongsTo `Client`
- Indice di unicità composto su (`clientId`, `normalizedName`)

### Modifiche alle API

- Nessuna API REST pubblica nuova obbligatoria.
- Server Actions per:
  - `createProjectAction` — crea una nuova commessa
  - `updateProjectAction` — aggiorna una commessa esistente
  - `closeProjectAction` — chiude una commessa (isActive = false)
- Funzioni server-side riusabili:
  - `listProjects()` — tutte le commesse per il backoffice
  - `listActiveProjects()` — solo commesse aperte per flussi operativi
  - `getProjectById(id)` — singola commessa con relazione client
- Validazioni: nome obbligatorio, cliente attivo, unicità nome per cliente, gestione errori con `ProjectDomainError`

### Modifiche al Frontend

- Aggiungere nav item "Commesse" nella sidebar del layout backoffice, usando l'icona `FolderKanban`.
- Creare pagina `dashboard/projects` con:
  - header descrittivo e CTA "Nuova commessa"
  - card KPI: totale commesse, attive, chiuse
  - lista strutturata con nome commessa, cliente associato, stato e azioni
- Creare componente `ProjectFormDialog` per creazione/modifica con:
  - campo nome commessa (input text)
  - selettore clienti attivi (select/combobox con `listActiveClients()`)
  - gestione errori e feedback utente
- Creare componente `ProjectActions` per edit e chiusura con conferma
- Applicare i vincoli di `docs/mockups/professional_coaching_planner/DESIGN.md`:
  - palette corporate blue per azioni primarie
  - superfici chiare con bordo leggero
  - tipografia Inter
  - spacing su base 4px/8px
  - badge stato con colore primario per "Aperta" e secondary/amber per "Chiusa"
